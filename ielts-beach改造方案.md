# 二、方案 A · 像素风重做（纯 CSS · 小像素格）

## A1. 核心原则
1. **统一像素单元**：定义 `--px`（如 `4px`），整个场景对齐到这个网格。**所有元素的每一个"格"都是 `--px × --px` 的实心方块**——这就是"小像素格构成"的落地方式。
2. **推翻一切平滑造型**：禁用 `border-radius`、`radial-gradient`、发光 `box-shadow`、`clip-path` 平滑多边形。
3. **有限调色板**：每个元素只用 2–4 阶颜色（亮面/主色/暗面/描边），不用连续渐变。
4. **跳帧动画**：`animation` 一律加 `steps(N)`，让海浪/棕榈/海鸥逐帧跳变（像素风动画是离散的，不是补间）。

## A2. 关键技法：box-shadow 像素图谱 + 生成器
纯 CSS 画精细像素图的标准做法——**一个 `--px` 大小的种子元素，用一长串 `box-shadow` 在网格坐标上"点"出每个像素格**。手写坐标太痛苦，所以配一个小生成器，让像素图维护起来像编辑"字符画"：

```js
// public/js/pixel-art.js —— 把字符画编译成 box-shadow
const PALETTES = {
  sun: { Y:'#FFE066', O:'#FFB02E', R:'#F08A24', '.':null },
};
export function compile(map, unit, palette) {
  const out = [];
  map.forEach((row, y) =>
    [...row].forEach((ch, x) => {
      const c = palette[ch];
      if (c) out.push(`${x*unit}px ${y*unit}px 0 0 ${c}`);
    })
  );
  return out.join(',');               // 赋给某元素的 box-shadow
}

// 太阳的像素图谱（8×8，可加密到 16×16 更精细）
const SUN = [
  '..YYYY..',
  '.YYOOYY.',
  'YYOOOORY',
  'YOOOORRY',
  'YOOOORRY',
  'YYOOOORY',
  '.YYOORY.',
  '..YYYY..',
];
```
CSS 侧：种子元素 `width/height: var(--px)`，`box-shadow` 由 JS 注入（或编译成 CSS 变量）。**改美术 = 改字符画地图**，直观、可版本管理、零素材文件。

## A3. 要画的像素图谱清单（全力期逐个补地图）
- ☀️ 太阳（8×8 或 16×16，带亮/暗面）
- 🌙 月亮（带环形山暗格做月相）
- ☁️ 云（2–3 阶白，硬边轮廓，多朵不同形）
- 🌊 海浪（一段可平铺的波纹 tile，`steps()` 横移）
- 🏖 沙滩（点状颗粒纹理，2 阶沙色抖动）
- 🌴 棕榈（树干分段 + 叶片，叶片 `steps()` 摇摆）
- 🐦 海鸥（2–3 帧扇翅 sprite，切帧飞行）
- ⭐ 星空（夜间，单格白点随机分布 + `steps()` 闪烁）

## A4. 天空/海面这种大面积怎么办（性能权衡 → 保持轻量）
- **大面积背景**（天空、海、沙底色）：用**纯色色带 + 2 色像素抖动**（`repeating-linear-gradient` 做硬边阶梯，或棋盘点 box-shadow），**不**逐像素 box-shadow（否则几千个 shadow 会卡）。
- **精细 sprite**（太阳/云/棕榈/海鸥）才用 box-shadow 图谱，每个几十~两百格，性能没问题。
- 这样既满足"小像素格构成"，又**不引入任何依赖、核心仍轻量**。

## A5. 调色板（重定义，像素友好）
把 `variables.css` 里的连续渐变色，改成**每元素几阶离散色**（日/夜各一套）。例：天空日 = 3 阶蓝（`#5BA9E6 / #7FC4F0 / #A9DCF5`）做阶梯，而非平滑渐变。字体已是 `Press Start 2P`（像素字体），UI 卡片也顺势改硬边、无圆角，整体统一。

## A6. 涉及文件 & 实施步骤
- 新增 `public/js/pixel-art.js`（生成器）
- 重写 `public/css/beach.css`（删渐变造型，改像素网格 + box-shadow 变量）
- 调整 `public/css/variables.css`（离散调色板 + `--px`）
- 微调 `weather.css`（雨雪也走像素粒子）+ `index.html`（种子元素结构）
- 步骤：① 落地 `--px` 网格 + 生成器 → ② 先做太阳/云验证观感 → ③ 补齐其余图谱 → ④ 动画切 `steps()` → ⑤ 日/夜+天气联调 → ⑥ 真机性能检查（降级开关）

# 三、方案 B · 大词库（综合三方案 · 够大且轻量）

## B1. 你要的"综合三者取优点"——分层架构
```
        ┌─────────────────────────────────────────┐
        │   words.js 只依赖一个抽象层 WordSource     │  ← 数据源可换，前端不动
        └───────────────────┬───────────────────────┘
        ┌───────────────────┼────────────────────┐
        ▼                   ▼                    ▼
 ① AI 预生成分片库      ② 词典 API 兜底       ③ D1 后端（未来位）
 （主力·离线·字段全）   （补全·无限扩展）     （同步·用户词本）
  GitHub/CF 托管 JSON   dictionaryapi.dev     部署后端后无缝接入
  + IndexedDB 缓存      按需·可开关           换 WordSource 实现即可
```

## B2. 三者各取什么优点

| 来源 | 取它的优点 | 角色 |
|------|----------|------|
| **① AI 预生成**（原方案A）| 字段全（音标/释义/例句/CEFR/雅思频率/同反义）、质量高、**离线可用** | **主力词库** |
| **② 词典 API**（原方案B）| 无限扩展、覆盖词库外的词、零维护 | **兜底/补全**（可开关，缺字段或查生词时才调）|
| **③ D1 后端**（原方案C）| 跨设备同步、用户自定义生词本 | **未来升级位**（接口预留，后端部署好再接）|

> 关键设计：`WordSource` 是一个**接口**（`getManifest() / loadPack(id) / lookup(word)`）。今天它的实现 = "静态分片 + 词典 API 兜底"；将来后端 D1 上线，只需把实现换成 D1 版，`words.js` 和 UI **一行不用改**。这就是"综合"的工程落点。

## B3. "够大"怎么保证
- **数据生产（离线一次性）**：以公开雅思高频词表 / 学术词表（AWL 570 词等）为词形清单，用 Claude 批量补全成完整字段，目标 **2000–3000 词**（够覆盖雅思核心）。
- 词典 API 兜底 → 理论可查**任意**英文词，词库上限实际无限。

## B4. "轻量"怎么保证（硬约束）
- 大词库**不打包进核心包**。切成分片（如 `pack-b1.json / pack-b2.json / pack-c1.json / pack-c2.json`，每片几百词）+ 一个 `manifest.json`（列分片、词数、版本）。
- **启动只拉 manifest + 第一片** → 立刻能用；其余**按需懒加载**（进入某级别 / 当前池快用完时再拉）。
- 下载过的分片存 **IndexedDB**，二次打开离线直读、不重复下载（靠 manifest 版本号判断更新）。
- 结果：核心 app 体积几乎不变，词库再大也不拖慢首屏。

## B5. 数据 schema
沿用现有字段（`id/word/phonetic/partOfSpeech/definition/cefrLevel/ieltsFrequency/exampleSentence/topicTags/synonyms/antonyms`），新增分片 `meta`（version/level/count）。

## B6. 涉及文件 & 实施步骤
- 改造 `scripts/build-words.js`：占位符 → 真管线（读词单 CSV → 调 AI 补全 → 校验去重 → 输出分片 + manifest）
- 新增 `public/js/word-source.js`（WordSource 抽象 + 静态分片实现 + 词典 API 兜底）
- 改 `public/js/words.js`：`_loadFromJSON` → 依赖 `WordSource`，加 IndexedDB 缓存
- 托管分片（GitHub raw 或你要部署的 Cloudflare Pages/R2）
- 步骤：① 定词单来源 → ② build 管线跑出分片 + manifest → ③ 托管 → ④ 前端接 WordSource + 懒加载 → ⑤ IndexedDB 缓存 → ⑥（可选）开词典 API 兜底 → ⑦ 留好 D1 升级接口
